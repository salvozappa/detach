# Installation Spec

## Overview

The installation script sets up a detach.it instance for a specific git repository. It generates required SSH keys, guides the user through deploy key setup, clones the repository, and starts the services.

**Design goals:**
- Single command to get started
- Clear, interactive guidance for deploy key setup
- Verify configuration before starting services
- Idempotent (safe to re-run)

---

## Usage

```bash
./install.sh <repository-url>
```

**Examples:**
```bash
./install.sh git@github.com:user/my-project.git
./install.sh https://github.com/user/my-project.git
```

---

## Installation Flow

### Step 1: Validate input

```
$ ./install.sh git@github.com:user/my-project.git

Detach Installation
======================

Repository: git@github.com:user/my-project.git
```

**Validation:**
- Repository URL is provided
- URL format is valid (SSH or HTTPS)
- HTTPS URLs are converted to SSH format (deploy keys require SSH)

**URL conversion:**
| Input | Converted |
|-------|-----------|
| `https://github.com/user/repo.git` | `git@github.com:user/repo.git` |
| `https://github.com/user/repo` | `git@github.com:user/repo.git` |
| `git@github.com:user/repo.git` | (unchanged) |
| `git@gitlab.com:user/repo.git` | (unchanged) |

### Step 2: Generate SSH keys

```
Generating SSH keys...
  ✓ Internal bridge key
  ✓ Repository deploy key
```

**Keys generated:**

| File | Purpose |
|------|---------|
| `keys/bridge` | Bridge → sandbox SSH (internal) |
| `keys/bridge.pub` | Public key for sandbox authorized_keys |
| `keys/deploy_key` | Sandbox → git remote (user's repo) |
| `keys/deploy_key.pub` | Public key user adds to GitHub/GitLab |

**Key generation commands:**
```bash
ssh-keygen -t ed25519 -f keys/bridge -N "" -C "detach.it-internal"
ssh-keygen -t ed25519 -f keys/deploy_key -N "" -C "detach.it"
```

**Idempotency:** If keys already exist, skip generation and notify user:
```
  ✓ Internal bridge key (exists)
  ✓ Repository deploy key (exists)
```

### Step 3: Display deploy key instructions

```
============================================================
ACTION REQUIRED: Add deploy key to your repository
============================================================

Copy this public key:

ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAI... detach.it

Add it to your repository:

  GitHub: Repository → Settings → Deploy keys → Add deploy key
  GitLab: Repository → Settings → Repository → Deploy keys
  Bitbucket: Repository → Settings → Access keys → Add key

⚠️  IMPORTANT: Enable "Allow write access" to push changes

Press Enter when you've added the key...
============================================================
```

Script pauses here until user confirms.

### Step 4: Verify repository access

```
Verifying repository access...
  Testing SSH connection... ✓
  Cloning repository... ✓
```

**Verification steps:**

1. **SSH connection test:**
   ```bash
   GIT_SSH_COMMAND="ssh -i keys/deploy_key -o StrictHostKeyChecking=accept-new" \
     git ls-remote <repo-url> HEAD
   ```

2. **If verification fails:**
   ```
   ✗ Could not access repository

   Common issues:
   - Deploy key not added yet
   - Write access not enabled
   - Wrong repository URL

   Retry? [Y/n]
   ```

3. **Clone repository:**
   ```bash
   GIT_SSH_COMMAND="ssh -i keys/deploy_key" \
     git clone <repo-url> /path/to/workspace
   ```

### Step 5: Generate authentication token

```
Generating authentication token...
  ✓ Token saved to data/auth_token
```

See [authentication.md](./authentication.md) for token spec.

### Step 6: Start services

```
Starting services...
  ✓ sandbox
  ✓ bridge
  ✓ webview

All services running.
```

**Command:**
```bash
docker-compose up -d
```

### Step 7: Display pairing information

```
============================================================

============================================================

  http://localhost:8080?token=a8f3b2c1d4e5f6...

```

## Configuration Files

### Generated files

| File | Purpose |
|------|---------|
| `keys/bridge` | Bridge SSH private key |
| `keys/bridge.pub` | Bridge SSH public key |
| `keys/deploy_key` | Deploy SSH private key |
| `keys/deploy_key.pub` | Deploy SSH public key |
| `data/auth_token` | Authentication token |
| `.env` | Runtime configuration |

### .env file

Generated by install script:

```bash
# Repository configuration
DETACH_REPO_URL=git@github.com:user/my-project.git

# Authentication
DETACH_TOKEN=a8f3b2c1d4e5f6...

# Optional: Web push (leave empty to disable)
VAPID_PUBLIC_KEY=
VAPID_PRIVATE_KEY=
VAPID_SUBJECT=
```

### .gitignore

The following should be ignored (not committed):

```gitignore
# Generated keys (sensitive)
keys/bridge
keys/bridge.pub
keys/deploy_key
keys/deploy_key.pub

# Runtime data
data/
.env

# Keep the directory structure
!keys/.gitkeep
!data/.gitkeep
```

## Docker Compose Changes

### Environment variables

```yaml
services:
  sandbox:
    environment:
      - DETACH_REPO_URL=${DETACH_REPO_URL}
    volumes:
      - ./keys/bridge.pub:/tmp/authorized_keys:ro
      - ./keys/deploy_key:/home/detach-dev/.ssh/id_ed25519:ro
```

### Sandbox entrypoint

The sandbox container should clone the repo on first start:

```bash
#!/bin/bash
# /entrypoint.sh

REPO_DIR="/home/detach-dev/workspace"

if [ ! -d "$REPO_DIR/.git" ]; then
  echo "Cloning repository..."
  git clone "$DETACH_REPO_URL" "$REPO_DIR"
fi

exec "$@"
```

## Re-running Installation

The script is idempotent:

| Scenario | Behavior |
|----------|----------|
| Keys exist | Skip generation, use existing |
| Repo already cloned | Skip clone, pull latest |
| Services running | Restart services |
| Different repo URL | Error: "Instance already configured for X. Use --force to reconfigure." |

### Force reconfiguration

```bash
./install.sh --force git@github.com:user/different-repo.git
```

This will:
1. Regenerate all keys (invalidates existing deploy key)
2. Remove existing workspace
3. Clone new repository

## Uninstallation

```bash
./uninstall.sh
```

**Actions:**
1. Stop and remove containers
2. Remove volumes (with confirmation)
3. Remove generated keys
4. Remove .env

```
This will remove all detach.it data including:
- Docker containers and volumes
- SSH keys
- Cloned repository

Are you sure? [y/N]
```

## Error Handling

### Invalid repository URL

```
Error: Invalid repository URL

Expected formats:
  git@github.com:user/repo.git
  https://github.com/user/repo

Run: ./install.sh <repository-url>
```

### Docker not running

```
Error: Docker is not running

Please start Docker and try again.
```

### Port already in use

```
Error: Port 8080 is already in use

Either:
  1. Stop the service using port 8080
  2. Set a custom port: PORT_WEB=9080 ./install.sh <repo>
```

### SSH key permission denied

```
Error: Could not access repository

The deploy key was rejected. Please verify:
  1. The key was added to the correct repository
  2. Write access is enabled (if you need to push)
  3. The repository URL is correct

Your public key:
ssh-ed25519 AAAAC3NzaC1... detach.it

Retry? [Y/n]
```

## Advanced Options

### Custom ports

```bash
PORT_WEB=9080 PORT_WS=9081 ./install.sh <repo>
```

### Custom branch

```bash
./install.sh <repo> --branch develop
```

### Skip interactive prompts

For automated setups (CI, scripts):

```bash
./install.sh <repo> --non-interactive
```

Assumes deploy key is already configured. Fails if verification fails.

## Security Notes

### Key permissions

Install script sets restrictive permissions:

```bash
chmod 600 keys/bridge keys/deploy_key
chmod 644 keys/bridge.pub keys/deploy_key.pub
chmod 600 data/auth_token
chmod 600 .env
```

### Deploy key scope

Deploy keys are scoped to a single repository. This is more secure than user-level SSH keys since:
- Compromise only affects one repo
- Can be revoked without affecting other repos
- Write access is explicitly opt-in

### Token in .env

The auth token in `.env` should be protected:
- File permissions: 600
- Not committed to git
- Regenerate if compromised: `./install.sh --regenerate-token`
